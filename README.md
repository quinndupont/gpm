# Volta — Teaching language models to revise poetry through synthetic education

<p align="center"><em>Encoding pedagogical process rather than poetic product</em></p>

Language models produce poetry that is formally competent but stylistically empty—lacking the distinctive voice that makes verse matter. This project implements **synthetic education**: a methodology that trains models not on poems but on *poetry instruction*—critique, revision dialogue, and craft lessons—generated by a frontier model in a consistent pedagogical persona.

A two-phase pipeline produces a structured corpus that fine-tunes separate **Educator** (mentor/critic) and **Poet** (generator) models, which operate in a multi-agent revision loop at inference. Both models (Llama 3.1 8B, QLoRA, Q4_K_M) run locally on consumer hardware. **RevFlux**, a process-oriented benchmark, measures revision dynamics rather than output quality. The trained system exhibits substantive critique-responsive revision, coarse-to-fine editing dynamics, and prompt-sensitive behaviour absent in the vanilla baseline—demonstrating that encoding pedagogical process rather than poetic product enables qualitatively different revision behaviour.

**The core insight:** The best poetry teachers have genuine voice and passionate opinions—they don’t function as neutral evaluators. Volta encodes this through a dual-model architecture where the Educator’s aesthetic judgments serve as the Poet’s loss function, mirroring the apprenticeship model that has produced great poets for centuries.

## What exists today

### Dual-model architecture

- **Educator model** — QLoRA fine-tuned Llama 3.1 8B with a distinctive critical persona. Trained on synthetic critique, cliché autopsy, comparative analysis, and craft lessons. Provides structured feedback with genuine aesthetic opinions rather than generic evaluation.
- **Poet model** (rhyme-trained, inference default) — Separate QLoRA fine-tuned Llama 3.1 8B trained on form-specific brief+poem+critique data plus a curated strong-rhyme corpus. Generates and revises poetry in response to Educator feedback.
- **Multi-agent revision loop** — At inference, the Educator critiques the Poet’s draft; the Poet revises; the Educator re-evaluates. This loop produces substantive revision dynamics absent from single-model generation.

### Training pipeline

- **Synthetic data generation** — Frontier model (Claude API) generates seed data in the Educator persona: critiques, comparisons, revision briefs, poet pairs, rhyme pairs. An interim Educator trained on seed data then generates additional training material (briefs, autopsies, lessons) locally, reducing API costs.
- **Contrastive learning** — 34k good poems contrasted with 114 bad examples. Bad amateur poetry serves as training data for cliché autopsy tasks: the Educator learns to dissect *why* clichés fail and what the poet was originally reaching for.
- **Deterministic form analysis** — Rhyme scheme scoring algorithm producing letter patterns (ABAB, CDCD) linked to specific poem types. CMU Pronouncing Dictionary-based rhyme detection, meter analysis, and form compliance checking.
- **Cloud-to-edge training** — Modal serverless GPUs for training → GGUF quantization (Q4_K_M) → local inference via llama.cpp on Mac Mini M4. Two ~4.5GB models fit comfortably in 24GB unified memory.

### RevFlux benchmark

A process-oriented benchmark measuring *how* a model revises rather than *what* it produces: per-line change magnitude, where change concentrates, how dynamics vary by prompt category and revision depth. Only the trained model runs the full revision loop; vanilla models produce initial drafts for comparison.

### Evaluation tools

- `rhyme_analyzer.py` — CMU-based rhyme detection: scheme identification, strict vs. slant classification
- `meter_analyzer.py` — Scansion via CMU: metrical pattern detection, consistency scoring
- `form_registry.py` — Form definitions (sonnet, villanelle, limerick, ghazal, etc.)
- `voice_consistency.py` — Educator output analysis: banned phrase detection, specificity scoring

## Where the project is headed

### Aesthetic Embedding Model (AEM)

The next major component: a contrastive-trained encoder that produces embeddings where proximity reflects **aesthetic kinship** rather than semantic similarity. Two poems about death by different quality levels should be far apart; two poems about different topics by the same master should be near each other. Off-the-shelf sentence embeddings cluster by topic, which is nearly useless for aesthetic evaluation.

The AEM will use a bi-encoder architecture (DeBERTa-v3-large or encoder layers extracted from larger open models) fine-tuned with Multiple Negatives Ranking Loss on same-poet pairs, same-movement pairs, and good/bad contrastive pairs from the existing dataset. Style-content disentanglement via adversarial decorrelation will separate *how* a poem is written from *what* it’s about. A pre-computed **Period Atlas** of literary movement centroids and covariance matrices will enable temporal fit analysis.

**Poems as trajectories:** Rather than embedding a poem as a single point, the AEM embeds each line separately, producing a trajectory through aesthetic space. The shape of this trajectory encodes structural information—volta, envelope structure, narrative arc—that a single-point embedding destroys. Trajectory features (displacement, velocity, angular acceleration, circularity, path signatures) become both visualization data and reward signals.

### Geometric visualization workshop

A web-based interface (Next.js 15, Vercel AI SDK) that makes visible what experienced poets sense intuitively—where a poem moves, where it stalls, where it sits in literary history, and how its ending transforms its beginning. Six visualization modes:

- **Trajectory View** — The poem as a path through projected style space, color-coded by velocity. Dead zones and volta candidates highlighted. Click any point for line-specific critique.
- **Velocity Profile** — The poem’s structural pulse. Form template ghosts show expected dynamics (where a sonnet’s volta *should* spike). Smoothing slider reveals both micro and macro structure.
- **Constellation Map** — The poem’s position among literary movements and reference poets. See your aesthetic ancestors and where your work lives in literary history.
- **Retroreading Heatmap** — How much each line’s meaning changes after reading the ending. Measures whether the poem retroactively transforms itself. High total retroreading = an ending that earns its weight.
- **Cliché X-Ray** — Phrase-level language freshness with four categories: fresh, worn, clichéd, and *transformed* (a cliché subverted or repurposed—craft, not failure).
- **Sonic Terrain** — Phonemic density, vowel register, and alliteration patterns per line. Entirely deterministic via CMU Pronouncing Dictionary.

The interface uses **progressive disclosure** across three skill levels: Beginner (cliché detection, sonic awareness, basic form feedback), Intermediate (trajectory analysis, constellation mapping), Advanced (retroreading, voice targeting, what-if branching with geometric consequences).

### Extended benchmark framework

Building on RevFlux, a six-layer benchmark system moving from deterministic to model-judged:

|Layer              |Measures                                            |Method                                       |
|-------------------|----------------------------------------------------|---------------------------------------------|
|Formal compliance  |Rhyme, meter, structure                             |Deterministic (exists today)                 |
|Linguistic craft   |Cliché density, image concreteness, lexical surprise|Lexical databases + base LLM perplexity      |
|Smell test         |Workshop-poem detection, MFA-voice detection        |Embedding distance from “safe poem” centroids|
|Structural         |Volta strength, trajectory shape, circularity       |AEM trajectory analysis                      |
|Retroreading       |How much ending transforms beginning                |Dual-pass embedding (causal + bidirectional) |
|Temporal aesthetics|Period fit, anachronism, zeitgeist gap              |Mahalanobis distance to period distributions |

### Scaling the models

Exploring larger open models for both Educator and Poet: LLaMA 3.1 70B, Mixtral 8x22B, DeepSeek-V2, Qwen 2.5 72B. The asymmetric architecture enables a larger Educator (70B, deeper aesthetic understanding) judging a smaller Poet (7–14B, fast generation). Educator judgments may be distillable into a smaller inference-time critic.

The Poet model’s training will expand to include REINFORCE with the Educator as reward model, with the rhyme scorer and trajectory quality metrics as additional reward signals—rewarding not just good language but good structural movement through the aesthetic space.

## Full pipeline

|Step|What                                                |Frontier API|Local         |
|----|----------------------------------------------------|:----------:|:------------:|
|1   |Hard tasks (critiques, comparisons, revision briefs)|✓           |—             |
|2   |Train interim educator on seed                      |—           |Modal GPU     |
|3   |Export + download interim educator                  |—           |Modal + local |
|4   |Local educator generates briefs, autopsies, lessons |—           |llama.cpp     |
|5   |Poet pairs                                          |✓           |—             |
|6   |Rhyme pairs + approval examples                     |✓           |—             |

Seed data (critiques, comparisons, revision briefs, poet pairs, rhyme pairs) is generated by a **frontier model** in the Educator persona. Briefs, autopsies, and lessons are then generated by the **interim educator** (trained on seed, run locally). The workflow also runs `generate_dialogues.py` and `generate_approval_examples.py` before the final prepare step. The **Poet** used at inference is the rhyme-trained poet (form-specific brief+poem+critique data + curated strong-rhyme corpus).

### 2. Prepare training data

```bash
python scripts/data_generation/prepare_training_data.py [--educator-only] [--poet-only] [--min-samples N] [--seed N]
python scripts/data_generation/prepare_rhyme_training_data.py [--general-frac 0.2]
```

Combines outputs into chat format. Rhyme data: curated strong-rhyme poems + form-specific pairs (80% rhyme / 20% general by default). `--min-samples N`: requires ≥N examples; if set, caps train/valid (quick test).

### 3. Upload to Modal

```bash
python scripts/modal/upload_data.py
```

Uploads `data/educator_training/`, `data/poet_training/`, and `data/rhyme_training/` to `poetry-data` volume.

### 4. Train + export (Modal)

```bash
# Educator
modal run scripts/modal/train_educator.py [--num-epochs-override N]
modal run scripts/modal/export_gguf.py::export_educator

# Poet (general) + Rhyme poet (inference default)

modal run scripts/modal/export_gguf.py::export_poet

# Or orchestrate
modal run scripts/modal/modal_app.py [--educator-only] [--poet-only] [--train-only] [--num-epochs N]
```

### 5. Download + inference

```bash
modal volume get --force poetry-gguf llama3.1-8b-educator-Q4_K_M.gguf models/
modal volume get --force poetry-gguf llama3.1-8b-poet_rhyme-Q4_K_M.gguf models/

python scripts/inference/pipeline.py "Write a poem about winter light" [--config PATH]
```

Default config uses the rhyme-trained poet. **Reference chat server:** `python serve_gpm.py [port]` (default 11435). `POST /api/chat` with JSON `{ "messages": [ {"role":"user","content":"..."} ] }` for streaming (application/x-ndjson).

## Design philosophy

This project challenges the dominant paradigm in AI poetry, which pursues epistemological superintelligence—models that *know more* about poetry. Volta instead pursues **pedagogical voice**: models that have genuine opinions, can articulate *why* something fails, and can teach through dialogue rather than evaluation.

The two-model architecture separates concerns that should be separate: critique and generation are different cognitive modes. The best human critics are not always the best writers. Specialization produces better performance on both tasks than a single model attempting both.

Bad poetry is not noise to be filtered—it is essential training data. The Educator’s cliché autopsy capability, trained on amateur poetry, is central to the system’s pedagogical value. Understanding *why* a phrase is dead and *what the poet was reaching for* is a more useful skill than simply identifying “good” and “bad.”

The long-term vision is an AI workshop that compresses what usually takes years of MFA-level education into a structured, interactive journey—not by replacing human teachers but by making the invisible geometry of poetry visible and manipulable.
